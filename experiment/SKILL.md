# Experiment

## 保存实验记录材料

运行实验需要保存包括但不限于以下**实验记录材料**，用于后续分析、复现、追溯。

- 实验原始数据日志（不得只保留整理后的表格！）
- 实验运行脚本
    - 运行benchmark的脚本（包含必要的环境变量、实验运行参数等）
    - 数据分析处理脚本
    - 绘图脚本
- 为了实验临时对项目修改的代码patch
- 实验总结文档
    - 整理后的图片、表格
    - 实验环境（机器配置、git commit）
    - 实验关键现象、结论
- ...

实验结果统一放在一个大的文件夹下 **请询问用户实验文件夹位置！**
在该文件夹下，按照实验类型建立多个子文件夹。每个实验可能会跑很多轮，为每一轮建立一个文件夹，将实验记录材料保存其中。

以下是一种*可能*的文件目录组织形式：

```
experiments/
    some-scripts.py
    exp1/
        SUMMARY.md
        draw-figure.py
        round1-datetime/
            code.patch
            raw_output_file.txt
            SUMMARY.md
            run.sh
        round2-datetime/
            ...
    exp2/
        SUMMARY.md
        raw_output_file_round1_datetime.txt
        raw_output_file_round2_datetime.txt
    ...
```

注意，

- 有些脚本、SUMMARY.md可能是所有实验共用的，有些则是某个实验的多轮运行共用的，有些则是某一轮实验独占的。
请你在做实验期间，根据需要重构experiments/目录。
- 有些实验可能比较简单，这时候你可以向上面的"exp2/"那样组织文件。

## 利用git维护实验脚本

除了实验原始数据不可修改外，中间产生的整理后的数据、数据分析、绘图脚本、实验总结报告都是可以修改的。
你需要维护实验目录的清晰整洁，不要简单地往文件夹中新增各类脚本、实验报告，最终导致整个目录充斥大量脚本，杂乱无章。

我推荐你使用**git**来管理实验材料。你可以将实验文件夹当成git项目来管理，每运行一轮新实验，创建一个git commit。这样你在修改过去的实验脚本、总结报告时，就不必担心修改了过去的实验脚本，导致过去实验无法复现了。

## 推荐的做事步骤

推荐你在做实验的时候，给自己列一个计划，一步一步执行，以下计划可供参考，实际可根据任务复杂程度修正计划。

1. 检查当前实验环境

> 机器资源是否被他人占用，以至于影响实验？注意，如果他人占用的资源不影响实验，可以继续执行；如果他人占用的资源影响了实验，请向用户报告，**严禁影响他人实验！**

2. 修改实验代码

3. 运行实验

4. 保存实验原始数据

5. 整理实验材料、数据分析、撰写实验报告、创建git commit

> 合理复用/修改过去的数据分析脚本、实验运行脚本，在过去的实验报告基础上撰写。

6. 恢复实验环境

> 实验临时修改的代码patch已经保存后，就可以在主项目仓库中恢复代码了。
